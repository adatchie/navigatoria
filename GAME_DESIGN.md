# 大航海時代 スタンドアローン版 — 概要仕様 & 実装計画

## 1. プロジェクト概要

### 1.1 コンセプト
大航海時代Online の仕様書を参考に、**スタンドアローンプレイ**の大航海時代ゲームを開発する。
16世紀の大航海時代を舞台に、プレイヤーは冒険家・商人・軍人として世界の海を航海し、
交易で富を築き、未知の土地や宝物を発見し、海上戦闘を繰り広げる。

### 1.2 技術スタック
| 項目 | 選定 |
|------|------|
| 言語 | TypeScript |
| UI フレームワーク | React 19 |
| 3D 描画 | Three.js (React Three Fiber) |
| 状態管理 | Zustand |
| データ永続化 | IndexedDB (Dexie.js) |
| ビルド | Vite |
| テスト | Vitest + React Testing Library |

### 1.3 規模感 (ミドルスケール)
- 開発期間目安: 2〜3ヶ月
- 原作の主要3システム（航海・交易・戦闘）を簡略化しつつ実装
- 発見・クエスト・スキルは段階的に拡張
- グラフィックス: シンプルな3D（低ポリゴン、手続き生成の海面）

---

## 2. オンライン→スタンドアローン変換方針

原作はMMORPGのため、以下のシステムを再設計する。

| 原作の要素 | スタンドアローン版での対応 |
|-----------|--------------------------|
| マルチプレイヤー交易 | NPC経済シミュレーション（需給・相場変動をAIが駆動） |
| 艦隊（他PC） | NPC仲間の雇用・AI操船による艦隊編成 |
| 商会 | 削除（将来的に商会NPCとして復活可能） |
| バザール（PC間取引） | NPC商人との直接取引に変換 |
| 大海戦（サーバーイベント） | タイムライン制イベントに変換（ゲーム内日数進行で自動発生） |
| 勅命クエスト | メインストーリー進行に統合 |
| チャット | NPC会話システムに変換 |
| 投資による同盟港 | プレイヤー単独の投資→影響度システムとして維持 |
| ランキング | 個人実績・記録システムに変換 |
| プライベート/パブリックモード | 不要（削除） |

## 3. ゲームシステム概要仕様

### 3.1 ワールド構成

原作に基づき、3つのステージに分割する。

```
┌─────────────────────────────────────┐
│              洋上 (3D)              │
│  ・世界地図ベースの航海フィールド     │
│  ・ゾーン分割（約30ゾーン）          │
│  ・風・潮流シミュレーション          │
│  ・災害イベント発生                  │
├─────────────┬───────────────────────┤
│    街 (3D)    │   陸地 (3D)         │
│  ・交易所     │  ・発見物探索        │
│  ・ギルド     │  ・遺跡探索         │
│  ・造船所     │  ・採集/調達        │
│  ・酒場       │  ・陸地戦闘         │
│  ・王宮       │                     │
│  ・銀行       │                     │
│  ・道具屋     │                     │
└─────────────┴───────────────────────┘
```

#### ワールドスケール（原作準拠）
- 全世界: 東西1,600km × 南北800km
- グリッド: 5m × 5m
- ゲーム内1日 = 実時間2分
- 船の標準サイズ: 全長30m

#### ゾーン構成（簡略化）
初期実装で約30ゾーン。ヨーロッパ、地中海、西アフリカ、東アフリカ、
インド洋、東南アジア、カリブ海などの主要海域をカバー。

### 3.2 プレイヤーキャラクター (PC)

#### 基本情報
- 名前（姓・名）
- 性別
- 出身国: イスパニア、ポルトガル、イングランド、フランス、ネーデルラント、ヴェネツィアの6ヶ国
- 外見: 体型、髪型、髪色、肌色から選択

#### 航海者レベル（3系統）
| レベル | 効果 | 上昇要因 |
|--------|------|----------|
| 冒険 | 冒険系スキル上限、船の使用条件 | 寄航、発見、冒険クエスト達成 |
| 交易 | 交易系スキル上限、船倉容量ボーナス | 交易、交易クエスト達成 |
| 戦闘 | 戦闘系スキル上限、船の使用条件 | 戦闘勝利、戦闘クエスト達成 |

- レベル上限: 50（段階的に上限解放イベント）
- 経験値計算: 必要経験値 = (現在レベル+1)² × 10
- 自分の職業系統以外の経験値は1/2

#### 職業システム
- 初期3職業: 見習冒険家、見習商人、見習軍人
- 転職: レベル条件を満たしてギルドで転職（原作の職業ツリーを簡略化して20〜30職種）
- 職業特典: 優遇スキルの熟練度50%減、専門スキルのランク+1

#### 名声・悪名・爵位
- **名声** (冒険/交易/戦闘): クエスト達成で獲得。NPC面会条件、装備条件、爵位条件に影響
- **悪名**: 海賊行為で増加。入港拒否、NPC艦隊に襲われるリスク
- **爵位**: 名声に応じて叙任。装備解禁、NPC特典

### 3.4 船システム

#### 船種（初期実装: 約20種）
- 大きさ: 小型、中小型、中型、中大型、大型の5段階
- タイプ: 帆船、漕船、漕帆船

#### 船の基本パラメータ
| パラメータ | 説明 |
|------------|------|
| 最大耐久力 | HPに相当。0で沈没 |
| 装甲 | 砲撃ダメージ軽減 |
| 最大容量 | 船員室+砲室+倉庫の合計 |
| 最大重量 | 超過すると速度低下 |
| 耐波性能 | 高波海域での耐久力低下抑制 |
| 旋回速度 | 回頭能力 |
| 縦帆性能 | 逆風時の速度に影響 |
| 横帆性能 | 順風時の速度に影響 |

#### 材質（9種類）
レッドパイン → ビーチ → エルム → チーク → オーク → マホガニー → 銅張り → ローズウッド → 鉄張り
- 材質により耐久修正(97-140%)、速度修正(95-110%)、価格係数が変化
- 街の発展度で使用可能材質が決まる

#### 容量・積載システム
- 船内を船員室・砲室・倉庫に分割
- 物資1個 = 容量0.1（10個で1樽）
- 改造でデフォルト値の50-150%範囲で調整可能
- 交易レベル補正: 最大容量 += 交易レベル × 2

#### 帆と速度計算
1. 帆の種類（縦帆=逆風に強い、横帆=順風で高速）を考慮
2. 風向・風速による速度係数を適用
3. 補助帆による修正
4. 材質による速度修正
5. スキル（操帆・操舵）修正
6. 積載重量による減速（最大重量の60%超で速度低下）
7. 潮流の影響

#### 船部品
| 部品 | スロット数 | 効果 |
|------|-----------|------|
| 補助帆 | マスト本数（最大6） | 速度向上、旋回低下 |
| 大砲 | 船側5+船首1+船尾1 | 砲撃攻撃 |
| 追加装甲 | (大きさ+3)/2 | ダメージ軽減 |
| 特殊兵装 | 1-2 | 衝突攻撃、白兵戦補助等 |
| 船首像 | 1 | 災害修正、疲労修正等 |
| 紋章 | 1 | 帆シンボル変更 |

### 3.5 航海システム

#### 洋上移動
- 3D海面上でリアルタイム操船
- 風向・風速が時間経過で変化
- 潮流が船を流す
- 天候変化（晴れ→曇→雨→嵐）

#### 航海日数経過処理
- 水・食糧を航海日数に応じて消費（船員数に比例）
- 船員の疲労度が日数とともに増加
  - 食糧・水が枯渇すると急増
  - 疲労度1000で船員減少開始→船員0で難破
- 船員の忠誠度が日数とともに低下
  - 給料未払いで急減
  - 忠誠0で叛乱リスク
- 船員の練度が航海日数に応じて増加

#### 特殊状態
- **壊血病**: 長期航海で発生。疲労上昇、船員減少
- **疫病**: 長期航海で発生。壊血病より深刻
- **叛乱**: 忠誠0で発生リスク。失敗すると難破
- **ネズミ**: 食糧が多い長期航海で発生。食糧減少

#### 災害イベント（洋上）
海域ごとに発生確率が設定された12種類のイベント:

| イベント | タイプ | 頻度 | 効果 |
|---------|--------|------|------|
| 霧 | 制約 | 高 | 視界減少 |
| 嵐 | 嵐 | 高 | 操船制約、沈没リスク |
| 凪 | 行動不能 | 中 | 風停止 |
| 座礁 | 行動不能 | 中 | 陸地近海で発生 |
| 藻 | 行動不能 | 中 | 低緯度で発生 |
| 氷山 | 障害物 | 小 | 衝突で大ダメージ |
| 大サメ | 障害物 | 小 | PC船に突進 |
| 大タコ/イカ | 障害物 | 小 | 海中から足が突き出る |
| 幽霊船 | 障害物 | 小 | 追跡してくる |
| 強い磁場 | 制約 | 小 | 羅針盤使用不可 |
| セイレーン | 制約 | 小 | 船員能力低下 |

### 3.6 交易システム

#### 交易品
- **20カテゴリ**: 食料品、酒類、調味料、嗜好品、香辛料、医薬品、繊維、染料、
  香料、雑貨、織物、貴金属、鉱石、宝石、装飾品、美術品、工芸品、武具、火器、工業品
- 各街に最大10の特産品
- 取引単位: 1, 2, 5, 10, 20, 50, 100, 200樽
- 重さにより積載制限

#### 価格決定
**購入価格** = 基本販売価格 × 販売相場% × パッケージ割引 + 交易税
- 相場は需給で変動（NPC経済シミュレーション）
- 流行度による乗数
- 交易税: 自国領0%、他国領5-14%、商館あり-8%

**売却価格** = 基本買取価格 × 買取相場%
- **名産品補正**: 文化圏距離 × 10% の上乗せ（遠くの特産品ほど高値）
- 必要交易品は20%増し

#### NPC経済シミュレーション（スタンドアローン版の肝）
- 各街でNPCによる売買が自動的に発生
- 需要・供給量が時間経過で変動
- プレイヤーの売買行動が相場に影響
- 戦争・疫病等の世界イベントが経済に影響
- 投資による街の発展度上昇 → 新交易品・船種解禁

### 3.7 発見システム

#### 発見物カテゴリ
| 種類 | 細分 | 発見方法 | 鑑定スキル |
|------|------|---------|-----------|
| 遺跡 | 史跡、宗教建築 | 視認 | 考古学/宗教学 |
| 宝物 | 歴史遺物、宗教遺物、美術品、財宝、化石 | 探索 | 考古学/宗教学/美術/財宝鑑定/生物学 |
| 生物 | 植物、虫類、鳥類、動物、海洋生物 | 生態調査 | 生物学 |

#### 発見フロー
1. クエスト受注 → 目標エリアへ航海
2. 発見用スキル使用（3秒後に判定）
3. スキルチェック: スキルランク+乱数5 >= 発見ランク+乱数5
4. 鑑定チェック: 鑑定スキルランク >= 発見物ランク
5. 宝物は開錠判定（危険度に応じて鍵・罠あり）
6. 発見成功: 冒険経験・名声獲得、発見履歴記録

### 3.8 洋上戦闘システム

#### 戦闘開始条件
- クエスト討伐対象
- NPC海賊との遭遇
- 危険海域での遭遇
- プレイヤーが仕掛ける（特定条件下）

#### 戦闘フロー
洋上マップ上でリアルタイム戦闘を行う。

**砲撃戦:**
1. 移動・操船で射程内に敵を捉える
2. 装填ゲージが80%以上で砲撃可能（100%で命中率UP）
3. 命中判定: 弾道学スキル vs 回避スキル → 命中/半命中/ハズレ
4. 被害計算: 貫通力 × 砲門数 - 装甲値 → 耐久度・船員数減少
5. 部分損傷: 帆・舵への損傷確率判定

**白兵戦:**
1. 船同士が接舷すると白兵戦
2. 船員数×(練度+剣術+装備) vs 船員数×(忠誠+防御+装備)
3. 船員数を削りあう
4. 船員0で拿捕

**戦闘終了:**
- 勝利条件: 敵全滅 or 退却
- 勝者: 戦闘経験、名声、戦利品（収奪スキル依存）
- 敗者: 難破 → 出港街に帰還（所持金・アイテム一部喪失）

#### NPC戦闘AI
- 作戦パターン: 砲撃重視、白兵重視、逃走
- 風上を取る行動
- 味方船との連携

## 5. マスターデータ設計（概要）

```typescript
interface Port {
```

## 6. 技術的考慮事項

### 6.1 3D描画方針
- **海面**: シェーダーベースの手続き生成（Gerstner波）
- **船**: 低ポリゴンモデル（GLTF形式）。初期は5-6種の基本モデル
- **街**: アイソメトリック風3Dビュー。建物は基本形状の組み合わせ
- **陸地**: 地形メッシュ+草木のインスタンス描画
- **パーティクル**: 砲撃エフェクト、波しぶき、天候
- **LOD**: 遠方の船は簡易表示

### 6.2 パフォーマンス
- ゲームループは `requestAnimationFrame` ベース
- 物理計算（風、潮流、船の動き）は軽量実装
- 3D描画は React Three Fiber のスケジュール機能活用
- NPC経済シミュレーションは Web Worker で並列実行

### 6.3 データ永続化
- IndexedDB にセーブデータを保存
- 複数セーブスロット対応
- 自動セーブ（街入港時、ゲーム終了時）
- セーブデータのエクスポート/インポート（JSON）

### 6.4 拡張性
- マスターデータは全てJSON
- モッド対応の余地を残す（カスタムデータ読込）
- イベントシステムはスクリプタブル設計

## 7. リスクと対策

| リスク | 影響 | 対策 |
|--------|------|------|
| 3Dアセット不足 | 見栄えが貧弱 | 手続き生成+シンプルなスタイルで統一感を出す |
| NPC経済のバランス | 交易が成立しない/簡単すぎる | パラメータをJSON化し、バランス調整を容易に |
| 戦闘の面白さ | リアルタイム操船が難しすぎる | ポーズ機能+スロー再生モードを用意 |
| コンテンツ量 | 初期は街・クエストが少ない | Phase2以降で段階的に追加。手続き生成クエストも検討 |
| ブラウザ性能 | 複雑な3Dシーンで重い | LOD、描画距離制限、低品質モードを用意 |
